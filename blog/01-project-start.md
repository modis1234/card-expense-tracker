# [카드 가계부 만들기 #1] 프로젝트 시작 - 왜 만들게 되었나?

## 들어가며

매달 카드 명세서를 받을 때마다 "이번 달은 어디에 돈을 많이 썼지?"라는 생각을 하게 됩니다. 카드사 앱에서 제공하는 통계는 있지만, 여러 카드를 사용하면 통합 관리가 어렵고, 내가 원하는 방식으로 카테고리를 분류하기도 힘듭니다.

그래서 직접 만들기로 했습니다. **카드 결제 내역을 자동으로 분류해주는 가계부 서비스**를요.

## 프로젝트 목표

### 핵심 기능
1. **카드사 엑셀 파일 업로드** - 신한, 삼성, 현대, KB, 롯데 등 주요 카드사 지원
2. **AI 자동 카테고리 분류** - OpenAI API로 거래 내역을 자동 분류
3. **통계 대시보드** - 카테고리별, 월별 지출 분석
4. **사용자 피드백 학습** - 수정한 카테고리를 학습해서 정확도 향상

### 왜 이 프로젝트인가?

**기존 서비스의 문제점:**
- 여러 카드사 통합 관리 어려움
- 카테고리 분류가 내 기준과 다름
- 수동 입력은 너무 귀찮음
- 데이터 소유권 (내 데이터를 내가 관리하고 싶음)

**이 프로젝트의 차별점:**
- 엑셀 파일만 업로드하면 자동 분류
- AI가 가맹점명을 보고 카테고리 추천
- 내가 수정한 내역을 학습해서 점점 정확해짐
- 오픈소스로 공개 예정

## 기술 스택 선정

### Frontend
- **Next.js 15 (App Router)** - 최신 React 프레임워크, SSR/SSG 지원
- **Tailwind CSS** - 빠른 UI 개발
- **React Query** - 서버 상태 관리

### Backend
- **NestJS** - TypeScript 기반, 구조화된 아키텍처
- **Prisma** - 타입 안전한 ORM
- **OpenAI API** - GPT-4로 카테고리 분류

### Database & Infrastructure
- **Supabase PostgreSQL** - 무료 티어, Auth 기능 제공
- **Vercel** - Frontend 배포 (무료)
- **Railway** - Backend 배포 (무료 티어)

### 왜 이 스택인가?

**비용 최소화:**
- 모든 서비스 무료 티어 활용
- 초기 사용자 수백 명까지 무료 운영 가능

**개발 생산성:**
- TypeScript로 전체 스택 통일
- Prisma로 타입 안전성 확보
- NestJS의 모듈 구조로 유지보수 용이

**확장성:**
- OpenAI → AWS Bedrock 전환 가능
- Supabase → 자체 DB 마이그레이션 가능
- 모노레포 구조로 모바일 앱 추가 용이

## 전체 로드맵

```
Phase 0: 기획 (2주) ────────────── 현재 진행 중
├─ 카드사 포맷 조사
├─ 카테고리 체계 설계
├─ DB 스키마 설계
└─ 와이어프레임 작성

Phase 1: Backend 개발 (3-4주)
├─ NestJS 프로젝트 초기화
├─ 파일 업로드 & 파싱
├─ AI 카테고리 분류
└─ REST API 구현

Phase 2: Frontend 개발 (3-4주)
├─ Next.js 프로젝트 초기화
├─ 인증 UI
├─ 거래 내역 관리
└─ 통계 대시보드

Phase 3: 통합 & 배포 (1-2주)
├─ E2E 테스트
├─ 버그 수정
└─ 프로덕션 배포

Phase 4: 모바일 앱 (선택적, 4-6주)
└─ React Native + Expo

Phase 5: 고도화 (지속적)
├─ Gmail API 연동
├─ 예산 관리
└─ 가족 계정
```

**예상 기간:** 2-3개월 (모바일 제외)
